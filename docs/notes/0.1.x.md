# Development Notes — 0.1.x

Working notes, context, and decisions for the **0.1.x** release series.
Newest entries first. This is dev-facing (the *why* and *how*); user-facing
changes live in [`../changelogs/0.1.x.md`](../changelogs/0.1.x.md).

## Key decisions

- **Packaging**: `src`-layout package built with the hatchling backend; the
  single console-script entry in `[project.scripts]` is the only entry point —
  no `main.py` / `__main__.py` (matches the `uv init --package` layout).
- **Versioning**: dynamic, sourced from git tags via hatch-vcs
  (`[tool.hatch.version] source = "vcs"`). Tagging `vX.Y.Z` is the single source
  of truth — no manual version bumps in `pyproject.toml`.
- **Release pipeline**: three least-privilege jobs
  (`build` → `publish-to-pypi` → `github-release`) so the PyPI `id-token` and
  repo `contents: write` scopes never coexist in one job.

## Gotchas

- CI checkout uses `fetch-depth: 0` — hatch-vcs needs full history and tags to
  derive the version; the default shallow clone would break versioning.
- PyPI Trusted Publishing requires a one-time "pending publisher" on PyPI
  (repo + workflow `release.yml` + blank environment) before the first release.
